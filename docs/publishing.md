# Publishing Process

This repository publishes packages from CI.

## Release preparation

Use the repository wrapper around Nerdbank.GitVersioning to prepare stable releases:

```powershell
git switch main
git pull --ff-only
git status --short
pwsh ./eng/prepare-release.ps1
```

Run the wrapper from a clean `main` only. Do not run `nbgv prepare-release` directly
for stable releases, or run either command from a feature or pull request branch.

The wrapper:

- runs `nbgv prepare-release` with the `release` settings in `version.json`;
- creates a `release/<version>` branch, for example `release/0.12.0`;
- bumps `main` to the next explicit development pattern, for example
  `0.13.0-dev.{height}`;
- converts only the release branch from a fixed three-part version such as `0.12.0`
  to the two-part version `0.12` and commits that change;
- returns to `main` with a clean working tree.

With the two-part release version, NBGV uses Git height as the patch component. The
conversion commit therefore produces `0.12.1`, and the next servicing commit produces
`0.12.2`.

Push the resulting `main` and `release/<version>` branches when the release is ready.
CI is responsible for build, test, pack, symbol packages, GitHub Release creation, and
NuGet publication.

## What triggers publishing artifacts

- CI runs on:
  - `pull_request`
  - `push` on `main`
  - `push` on `release/**`
- On pushes to `main` or `release/**`, the GitHub Release job runs after build, test,
  pack, and documentation lint succeed.

## Branch strategy

- `main`: prerelease flow (`-dev.*`).
- `release/*`: release branch flow (same packaging pipeline, intended for release stabilization).
- Branch name format can be `release/<target>` (for example: `release/0.9` or `release/1.0`).
- Release branches should be created from `main` with `eng/prepare-release.ps1`.

## Versioning (`version.json`)

- Versioning is handled by Nerdbank.GitVersioning.
- `version.json` is **never updated automatically** by CI.
- The current `main` prerelease pattern is:
  - `"version": "0.12.0-dev.{height}"`
- `eng/prepare-release.ps1` updates the version locally and commits the branch-specific
  changes. `main` keeps the explicit `-dev.{height}` pattern, while release branches use
  a two-part `major.minor` version so Git height supplies the stable patch number.
- `publicReleaseRefSpec` already includes `main` and `release/*`, so package and release
  versions generated by CI do not include the `.g<sha>` suffix.

### When should I modify it?

- Usual flow:
  - On `main`, keep the explicit prerelease pattern, for example
    `0.12.0-dev.{height}`.
  - Use `eng/prepare-release.ps1` to create a stable release branch and bump `main`.
  - On `release/*`, keep a two-part version such as `0.12`; each new commit then advances
    the stable patch (`0.12.1`, `0.12.2`, and so on).
- For a release branch created before the wrapper existed, run `nbgv set-version <major.minor>`
  once and commit the resulting `version.json` change.
- Do not replace the explicit development pattern on `main` with a two-part version.

## Do I need to create a Git tag manually?

- No for normal flow.
- On push to `main`/`release/*`, CI creates a GitHub Release with tag `v<NuGetPackageVersion>`.
- Only create tags manually for exceptional/manual workflows.

## NuGet publish status

- Package and symbol packages (`.snupkg`) are produced by `Build, Test, Pack` and published
  by the release job.
- `Publish to NuGet` runs after GitHub Release creation when the preceding step succeeds
  (`if: success()`).
- The `NUGET_API_KEY` repository secret must remain configured. Do not expose its value or
  attempt to validate it locally.
